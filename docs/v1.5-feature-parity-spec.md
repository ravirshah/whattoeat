# v1.5 → v2.0 Feature Parity Spec

## TL;DR

- **Port**: AI recipe generation, recipe modification, health-doc-driven goal setup, nutrition insights, and recipe favorites — all re-expressed as LLM calls rather than form UIs.
- **Drop**: Weekly calendar planner, meal prep scheduler, store-layout grocery optimizer, PDF export, recipe history browsing — all pure CRUD scheduling UI with no signal driving them.
- **Quick win 1**: AI Recipe Modification → surfaces inside Feed Me detail card (1 LLM call, no new route). S effort.
- **Quick win 2**: Health Document → Goal bootstrapping → folds into Onboarding as an optional "upload your labs" step that pre-fills macro targets. S–M effort.
- **Quick win 3**: Nutrition Insights (weekly summary + variance commentary) → ProactiveBrief card already shows today's macros; extend with an LLM-generated one-liner insight based on week's Feed Me history. S effort.

## Reimagined for v2.0

### R1 — Nutrition Insight (Weekly Variance Commentary)

**v2.0 surface**: `/home` ProactiveBrief card  
**Driving signal/LLM call**: Server-side aggregation of the current week's Feed Me recommendations that were accepted (from `recommendation` repo) + today's check-in energy level → one LLM call generates a 1-sentence insight ("You're 40g protein short this week; tonight's chicken option covers it").  
**UX**: The ProactiveBrief card already shows today's macros ring. Below it, a single sentence insight appears — no chart, no modal. Tapping it deep-links to the Feed Me page pre-filtered to high-protein options.

---

### R2 — AI Recipe Modification

**v2.0 surface**: `/feed-me` detail card (inline, below the recipe rationale)  
**Driving signal/LLM call**: User taps "Tweak it" on a Feed Me detail card → text input appears → single Gemini call with original structured recipe + modification prompt → updated structured recipe replaces the card in place.  
**UX**: No modal, no page navigation. The card morphs. A "Save tweaked" button appears if the user wants to persist the modified version to `/recipes/saved`. One LLM call, zero new routes.

---

### R3 — Health Document → Goal Bootstrapping

**v2.0 surface**: `/onboarding` step (optional step added after ConfirmTargetsStep, or accessible from `/profile/edit`)  
**Driving signal/LLM call**: User uploads a PDF (blood panel, DEXA, InBody) → server-side Gemini call extracts health markers → second LLM call maps markers to macro target adjustments and dietary flags → pre-fills the goal profile, presenting diffs vs the user's manually-set targets with a confidence score.  
**UX**: During onboarding, after setting manual goal targets, a card offers "Upload your labs for smarter targets." Upload triggers the two LLM calls and shows a diff: "Based on your A1C and LDL, I'd reduce carbs by 20g and flag saturated fat. Accept changes?" One confirm tap updates the profile. Available again from `/profile/edit`.

---

### R4 — Recipe History as Feed Me Context Signal

**v2.0 surface**: `/feed-me` recommendation engine (server-side, invisible to user as a UI feature)  
**Driving signal/LLM call**: The `recommendation/actions.ts` Gemini prompt already accepts context. Extend it to pull the last 7 days of saved + cooked recipes from the `recipes` repo and inject them as a "recently cooked, avoid repeating" context block in the prompt.  
**UX**: Users just notice Feed Me stops repeating the same meals. No list-browsing UI needed. The "last cooked" field already surfaces on `/home` ProactiveBrief. This is a zero-UI change that makes the engine meaningfully smarter.

---

### R5 — NL Nutrition Input → Pantry-aware Logging

**v2.0 surface**: `/pantry` NL input (extend existing parse-freeform server action)  
**Driving signal/LLM call**: Today, `parse-freeform.ts` categorises items into protein/produce/grain etc. Extend the same LLM call to also accept free-text nutrition entries ("just had 2 scoops protein + oats") and return both pantry deductions *and* a macro delta for the day. The macro delta gets written to a daily log table.  
**UX**: The Pantry NL input field hint changes to "Add items or log a meal…". When the LLM detects a nutrition entry (vs a stock item), it responds "Logged: 480 cal, 45g protein" instead of showing a chip. No second UI surface needed.

---

## Implementation Order

1. **R4 — Recipe History as Feed Me context signal** (S) — Pure server-side prompt augmentation. No UI changes. Immediate improvement to Feed Me quality with ~20 lines of code in `recommendation/actions.ts`.

2. **R2 — AI Recipe Modification in Feed Me** (S) — Add a "Tweak it" expander to the existing Feed Me `MealCard` detail view. One new server action, one Gemini call, inline UX. No new route.

3. **R1 — Nutrition Insight in ProactiveBrief** (S–M) — Requires a weekly aggregation query over accepted recommendations + a new LLM call for the insight sentence. Most of the data plumbing exists in the `recommendation` repo already.

4. **R3 — Health Document → Goal Bootstrapping** (M) — New optional onboarding step + `/profile/edit` entry point. Two LLM calls (extract markers, map to macros). Needs a file upload UI component and a new server action. High perceived value for engaged users.

5. **R5 — NL Nutrition Input → Pantry-aware Logging** (M) — Extends existing `parse-freeform` action and needs a new `daily_log` table in the schema. Moderate schema + server work; minimal UI change. Unlocks the macro ring on `/home` being populated without manual entry.

## Drop Rationale

| Dropped Feature | Why |
|----------------|-----|
| **Weekly Calendar Planner** | A 7-day scheduling grid is a legacy meal-tracker pattern. v2.0's posture is on-demand ("Feed Me now") not pre-scheduled. Adds enormous surface area with no AI signal driving slot assignments. |
| **Meal Prep Scheduler** | Deterministic rule-based logic (Sunday + Wednesday prep) dressed up as a feature. No user signal, no LLM, no personalisation. |
| **Smart Grocery List** | Derived from the weekly plan (dropped). A future "generate shopping list from my saved recipes" feature could be valuable but only after v2.0 has enough recipe data. Not now. |
| **Store Layout Manager** | Aisle mapping and route optimization is a logistics app, not a nutrition AI. Pure CRUD with zero intelligence payoff. |
| **PDF / HTML Export** | Power-user edge case. No AI signal. The only users who need a PDF are the same ones who would use a spreadsheet. Not blocking any core job-to-be-done. |
