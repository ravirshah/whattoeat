# WhatToEat 2.0 — Design Spec

**Date:** 2026-04-26
**Status:** Approved (sections 1–8) — pending implementation plan
**Author:** Ravi + assistant

---

## 1. Scope & Non-Goals

### Mission

WhatToEat 2.0 is a clean rebuild of the v1 pantry-recipe app as a personal meal-decision engine. Given a persistent pantry, a goal-oriented profile, and a daily check-in, the app answers one question on tap: **what should I eat right now?** It's architected so the eventual health platform (Apple Health / Eight Sleep / Superpower Labs / workouts) plugs in as additional signal without rewriting the core.

### In scope for v2.0

- Clean rebuild on Supabase (Postgres + Auth + Storage + RLS) + Next.js App Router on Vercel. v1 code and data are discarded.
- **Persistent pantry** (items-only, no quantities/expiry). Available-now toggle, LLM-assisted name normalization, real voice input.
- **Profile + goals.** Goal (cut/maintain/bulk), TDEE-suggested macro targets (editable), height/weight/sex/activity, allergies, dislikes, cuisines, equipment.
- **Daily check-in** (3 taps, ~5s): energy 1–5, training (none/light/hard), hunger (low/normal/high). Optional. Idempotent per day.
- **"Feed Me" decision flow.** Tap → 1–3 streamed meal candidates with rationale; expand to a full recipe; save and/or log a cook.
- **Recipe view** with structured ingredients/steps, animated `MacroRing`, pantry-coverage chips per ingredient.
- **Saved recipes + cooked log** with ratings.
- **Hexagonal core** (`src/engine/`) with `recommend(ctx)` as the only public surface; pure, testable without Next/Supabase/network.
- **LLM provider abstraction.** Default: latest Gemini, model selected by an eval harness at build time.
- **Sensor scaffolding.** Typed `SignalProvider` port and an empty `signals` slot on the engine context. No real provider implementations in v2.0.
- **Apple-discipline + Stripe-feel frontend.** Custom shadcn fork, design-token system, multi-tone palette, gradients as first-class, signature motion moments, brand-voice copy.
- **Worktree-friendly, agent-friendly project structure.** Frozen contracts, file-ownership rule, per-feature folders, Cline Kanban tasks per track, automatic agent-context bootstrap.
- **Two-layer code review** (subagent during work + CI/PR gates) and observability via Sentry + structured `recommendation_runs`.

### Explicit non-goals for v2.0

- Stripe billing / paywall enforcement.
- Native iOS app, photo-of-the-fridge, barcode scanning.
- Apple Health / Eight Sleep / Superpower data ingestion (architectural hooks only).
- Social, sharing, multi-user households.
- Workout module.
- Migrating any v1 Firestore data.

## 2. High-Level Architecture

Single Next.js App Router app with a hexagonal core. Three layers:

- **UI layer** (`src/app/**`, `src/components/**`) — Server Components for fetches, Client Components for interactive surfaces. No business logic.
- **Application services** (`src/server/**`) — Supabase access via Drizzle, request validation, context assembly. The only place where DB and engine meet.
- **Engine** (`src/engine/**`) — pure TypeScript. Zero imports from `next`, `@supabase/*`, `drizzle-orm`, or any network/DB package.

```
┌──────────────────────────────────────────────────────────────┐
│  Next.js App Router (Vercel)                                 │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ UI layer — RSC + Client Components                     │  │
│  │   pantry, profile, check-in, "feed me", recipe, saved  │  │
│  └─────────────────┬──────────────────────────────────────┘  │
│  ┌─────────────────▼──────────────────────────────────────┐  │
│  │ Server Actions + Route Handlers                        │  │
│  │   - mutations via Server Actions (Zod-validated)       │  │
│  │   - /api/recommend (streaming Route Handler)           │  │
│  │   - /api/integrations/* (sensor webhooks, future)      │  │
│  └─────────────────┬──────────────────────────────────────┘  │
│  ┌─────────────────▼──────────────────────────────────────┐  │
│  │ Application services  src/server/                      │  │
│  │   pantryService, profileService, checkinService,       │  │
│  │   recipeService, recommendationService                 │  │
│  └────────┬───────────────────────────────────┬───────────┘  │
│           │                                   │              │
│  ┌────────▼─────────────┐         ┌───────────▼──────────┐  │
│  │ Supabase (Postgres)  │         │ engine/  (pure core) │  │
│  │  RLS on every table  │         │  recommend(ctx)      │  │
│  └──────────────────────┘         │  Ports: LlmClient,   │  │
│                                   │   SignalProvider,    │  │
│                                   │   Logger             │  │
│                                   └──────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

### Engine public surface

```ts
// src/engine/recommend.ts
export async function recommend(
  ctx: RecommendationContext,
  deps: { llm: LlmClient; logger?: Logger }
): Promise<RecommendationResult>;
```

### Why this shape

- Engine is unit-testable in isolation. A fake `LlmClient` returns canned JSON; tests run fast with no env or network.
- LLM swap = one adapter file. Sensor add = one provider file. UI changes never touch the engine.
- When the bigger health platform arrives, `src/engine/` lifts cleanly into a shared `packages/engine/` without rewriting consumers.

## 3. Data Model

Postgres on Supabase. Drizzle ORM is the single source of truth; migrations via `drizzle-kit`. RLS on every table — default policy is `auth.uid() = user_id` for select/insert/update/delete unless noted.

### Tables

**`profiles`** — 1:1 with `auth.users`.
- `user_id` uuid PK, references `auth.users`
- `display_name` text
- `goal` text — `cut | maintain | bulk`
- `target_kcal` int, `target_protein_g` int, `target_carbs_g` int, `target_fat_g` int
- `height_cm` numeric, `weight_kg` numeric, `birthdate` date, `sex` text, `activity_level` text
- `allergies` text[], `dislikes` text[], `cuisines` text[], `equipment` text[]
- `created_at`, `updated_at` timestamptz

**`pantry_items`** — items only; no quantities or expiry in v2.0.
- `id` uuid PK, `user_id` uuid FK
- `name` text NOT NULL — normalized form ("chicken breast")
- `display_name` text — as the user typed it
- `category` text — `protein | produce | grain | dairy | pantry | other`
- `available` bool default true
- `added_at` timestamptz
- Unique: `(user_id, lower(name))`

**`checkins`**
- `id` uuid PK, `user_id` uuid FK
- `date` date
- `energy` smallint (1..5), `training` text (`none | light | hard`), `hunger` text (`low | normal | high`)
- `note` text
- Unique: `(user_id, date)`

**`recipes`**
- `id` uuid PK, `user_id` uuid FK
- `title` text, `description` text
- `ingredients` jsonb — Zod-validated `[{ name, qty, unit, note }]`
- `steps` jsonb — `[{ idx, text, durationMin? }]`
- `macros` jsonb — `{ kcal, protein, carbs, fat }`
- `servings` int, `total_minutes` int, `cuisine` text, `tags` text[]
- `source` text — `ai-generated | user-saved | imported`
- `generated_run_id` uuid FK → `recommendation_runs.id` (nullable)
- `saved` bool — true only if user explicitly favorited
- `created_at`, `updated_at`

**`cooked_log`**
- `id` uuid PK, `user_id` uuid FK, `recipe_id` uuid FK
- `cooked_at` timestamptz, `rating` smallint, `note` text

**`recommendation_runs`** — audit log of every "Feed Me" call.
- `id` uuid PK, `user_id` uuid FK
- `context_snapshot` jsonb — full input context at call time, immutable
- `candidates` jsonb — output `MealCandidate[]`
- `model` text, `prompts_version` text
- `prompt_tokens` int, `completion_tokens` int, `latency_ms` int
- `created_at` timestamptz
- Client-side: read-only. Inserts via service-role from Route Handler only.

**`signal_snapshots`** — scaffolded but empty in v2.0.
- `id` uuid PK, `user_id` uuid FK
- `source` text — `apple_health | eight_sleep | superpower`
- `kind` text — `sleep | hrv | training | labs | …`
- `payload` jsonb, `observed_at` timestamptz, `created_at` timestamptz
- Client-side: read-only. Writes via verified webhook with service-role.

### Indexes

- `pantry_items(user_id, available)`
- `checkins(user_id, date desc)`
- `recipes(user_id, saved, created_at desc)`
- `cooked_log(user_id, cooked_at desc)`
- `recommendation_runs(user_id, created_at desc)`

### Notes

- Pantry name normalization is LLM-assisted on insert; the unique constraint dedupes by lowercased normalized name. v1's "chicken / Chicken / chick3n" coexistence problem is gone.
- Recipes are stored as structured jsonb only — never free-text. The LLM produces structured JSON; we parse once and persist once. The v1 regex parser is gone.
- No `usage_stats` table. If/when paywall returns, it'll be a Stripe-driven `subscriptions` table + a route-handler gate, not an in-doc counter.

## 4. Decision Engine Contract

Lives in `src/engine/`. Pure TS. No imports from `next`, `@supabase/*`, `drizzle-orm`, or any network/DB package.

### Public types

```ts
type RecommendationContext = {
  pantry: PantryItem[];          // available=true subset only
  profile: Profile;
  checkin?: Checkin;             // today's, optional
  signals?: HealthSignals;       // EMPTY in v2.0; shape committed
  request: {
    mealType?: 'breakfast' | 'lunch' | 'dinner' | 'snack' | 'any';
    timeBudgetMin?: number;
    candidateCount: number;      // default 3
  };
};

type HealthSignals = {
  sleep?: { lastNightHours: number; quality?: 'poor' | 'ok' | 'great' };
  training?: { yesterdayLoad?: 'rest' | 'light' | 'hard'; muscleGroups?: string[] };
  recovery?: { hrvMs?: number; restingHr?: number };
  labs?: { fastingGlucose?: number; recentBiomarkers?: Record<string, number> };
};

type RecommendationResult = {
  candidates: MealCandidate[];
  rationale: string;
  modelUsed: string;
  tokens: { prompt: number; completion: number };
  latencyMs: number;
};

type MealCandidate = {
  title: string;
  oneLineWhy: string;
  ingredients: Ingredient[];
  steps: Step[];
  estMacros: Macros;
  servings: number;
  totalMinutes: number;
  cuisine?: string;
  tags?: string[];
  pantryCoverage: number;        // 0..1
  missingItems: string[];
};
```

### Ports (interfaces in the engine, no implementations)

```ts
interface LlmClient {
  generateStructured<T>(args: {
    system: string;
    user: string;
    schema: ZodSchema<T>;
    cacheKey?: string;
  }): Promise<{ value: T; tokens: TokenUsage; modelUsed: string }>;
}

interface SignalProvider {
  source: 'apple_health' | 'eight_sleep' | 'superpower';
  getSignals(userId: string, range: DateRange): Promise<Partial<HealthSignals>>;
}

interface Logger {
  info(...args: unknown[]): void;
  warn(...args: unknown[]): void;
  error(...args: unknown[]): void;
}
```

### Adapters (live OUTSIDE the engine, in `src/server/adapters/`)

- `GeminiLlmClient` — wraps the latest Gemini SDK. Structured output via response schema. Built-in 1-retry on schema-validation failure.
- `NullSignalProvider` — returns `{}`. The only `SignalProvider` shipped in v2.0.
- Future: `AppleHealthProvider`, `EightSleepProvider`, `SuperpowerProvider`.

### Engine internals (`recommend()` flow)

1. **Prompt build** (`engine/prompt.ts`) — deterministic JSON serialization of context, stable system prompt versioned (`PROMPTS_VERSION`).
2. **Plan call** (cheap model, e.g. Gemini Flash-tier): produce N candidate concepts — `[{ title, oneLineWhy, cuisine, est_minutes, pantryFit }]`. Strict schema.
3. **Detail calls** (parallel, one per chosen candidate): expand to full recipe — ingredients/steps/macros. Strict schema.
4. **Score & re-rank** (`engine/score.ts`) — pure function. Re-rank by pantry coverage, allergy safety (drop on hit), goal alignment (macros within ±30%).
5. **Sanitize** — defensive allergen filter, max-length enforcement, dedup.
6. **Return** with usage + latency.

### Error contract

- `EngineParseError` — schema validation failed after retry.
- `EngineSafetyError` — all candidates dropped on allergen filter.
- `EngineTimeoutError` — LLM exceeded the configured timeout.
- All thrown to the application layer; Route Handler maps to status codes.

### Eval harness (`src/engine/eval/`)

- ~20 frozen `RecommendationContext` fixtures.
- Rubric: schema validity, allergen safety, macro proximity, pantry-coverage realism, qualitative grade via a stronger judge model.
- CI script `bun run engine:eval` posts a cost-vs-quality table on PRs touching engine or adapters. **The first PR after the engine lands picks the production model from this data.**

## 5. Design System & Frontend

### Visual language

- **Apple-discipline + Stripe-feel.** Apple = the discipline (token system, motion physics, restraint where it earns it). Stripe = the feel (rich palette, confident gradients, editorial moments, personality in voice and motion).
- **Multi-tone palette, not single-accent.** A primary brand hue + 2 supporting hues (a warm and a cool) used systematically — warm for "fuel/energy" surfaces (training-day, calorie tiles), cool for "recovery" surfaces (sleep, rest-day), primary for action.
- **Gradients are first-class.** Animated mesh on home hero, gradient sweeps on the "Feed Me" CTA, gradient fills on `MacroRing`. Used confidently, not nervously.
- **Editorial layouts on marketing-y surfaces** — home hero, onboarding, empty states. Uniform card grids only inside the app's working surfaces.
- **Depth via subtle 3D.** Layered soft shadows, occasional inner-glow on hero elements, perspective transforms reserved for "Feed Me" reveal.
- **Brand voice.** Microcopy reads like a sharp, confident friend who cooks, not generic SaaS.
- **Type with character.** Geist Sans for UI text, Geist Mono for numerals (macros, timers). Display sizes get tighter tracking and heavier weights.
- **Color is composed, not rationed.** A curated 6-color category palette (protein/produce/grain/dairy/pantry/other) tuned to sit harmoniously together.
- **Motion is physical.** Spring defaults (`stiffness ~300, damping ~30`); never linear easings; respect `prefers-reduced-motion`.
- **Content max-width discipline.** Reading content caps around 640px on desktop. Not a dashboard.
- **Density is generous by default.** Spacious feels confident; cramped feels SaaS.

### Tokens (single source of truth)

- `src/styles/tokens.css` — CSS variables for color (`--surface`, `--surface-elevated`, `--text`, `--text-muted`, `--accent`, `--accent-fg`, `--ring`, `--ok`, `--warn`, `--err`), radius (`--r-sm/md/lg/xl`), shadow (`--shadow-1/2/3` — soft, layered), motion (`--ease-spring`, `--dur-snap`, `--dur-base`).
- `tailwind.config.ts` reads from tokens. **No raw hex/oklch in components — only `bg-surface`, `text-muted`, `rounded-lg`, etc.**
- Dark mode is a token swap; both modes designed simultaneously.

### Component strategy

- Fork shadcn primitives once into `src/components/ui/*`. **They are ours from day one** — not pinned to upstream, not auto-updated.
- **Sonner** for toasts, **Vaul** for mobile drawers/sheets, **cmdk** for the (later) command palette.
- **Lucide icons** locked at `stroke-width: 1.75` globally for consistency. ~5 brand icons drawn manually.

### Custom feature components

- `PantryChip` — togglable, with check state.
- `MacroRing` — Apple-Activity style three-ring with gradient fills, animated count-up, soft glow on target hit.
- `StatTile` — number-first, mono numerals, label below.
- `MealCard` — title, why-line, macros, pantry-coverage chip, time. The recommendation centerpiece.
- `SegmentedControl` — iOS-style for goal/training pickers.
- `CheckinSheet` — Vaul drawer for the 3-tap daily check-in.

### Signature moments (where personality shows up most)

- **Home hero** — animated gradient mesh, large display headline, "Feed Me" CTA as the centerpiece.
- **"Feed Me" reveal** — button compresses → gradient bloom → cards stagger in with spring physics → rationale fades in beneath.
- **`MacroRing`** — animated count-up numerals; soft glow on target hit.
- **Recipe card press** — subtle parallax on title/macros against the card surface.
- **Empty states** — short, witty, with a small animated illustration.
- **Onboarding** — feels like the start of a product, not a form. Each pane has motion + a hero visual.

### Screens for v2.0

1. **Auth** — Supabase magic-link, single screen.
2. **Onboarding** (3 panes): welcome → goal → seed pantry.
3. **Home** — greeting, today's check-in tile, "Feed Me" CTA, last 2–3 recommendations.
4. **Pantry** — chip grid, search, voice + paste add, normalization confirmation.
5. **Profile** — goals, macros, allergies, dislikes, cuisines, equipment, "Connections" placeholders for HealthKit/Eight Sleep/Superpower.
6. **Recommendation result** — streamed; rationale + 1–3 `MealCard`s; tap to expand.
7. **Recipe view** — full ingredients/steps, `MacroRing`, "I cooked this" + rating.
8. **Saved** — favorites + recently cooked, segmented.

### Layout

- **Mobile-first.** Bottom tab bar (Home / Pantry / Profile / Saved) with a center FAB-like slot for "Feed Me."
- Desktop = same single-column content with a sidebar nav. **Not a dashboard layout.**

### Accessibility (non-negotiable)

- WCAG AA on text and interactive contrast in both modes.
- Every interactive element keyboard-reachable; visible focus rings via `--ring`.
- Motion respects `prefers-reduced-motion`.
- Semantic HTML; ARIA only where the primitive doesn't already provide it.

### PWA

- Manifest + icons + a thin offline shell.
- Pantry view is offline-readable (cached SWR). Recommendations require network — graceful empty state when offline.

### Skills used during implementation (per the `/impeccable` workflow)

1. **`teach-impeccable`** — runs first; captures design language; locks accent, palette, voice.
2. **`frontend-design`** — initial component pass.
3. **`typeset` / `colorize` / `arrange`** — per-screen passes.
4. **`bolder`** — applied to home + onboarding + empty states to push past safe defaults.
5. **`animate`** — Feed Me flow, `MacroRing`, card reveals.
6. **`delight`** — applied to **core pages**: home, the Feed Me reveal, recipe view, daily check-in, onboarding final pane, "I cooked this" flow.
7. **`polish` / `normalize` / `audit`** — final pass before ship.
8. **`quieter`** — kept on hand only if a surface tips into busy.

## 6. Parallelization & Workflow

The repo is structured so independent tracks of work touch disjoint file sets. Subagents can be dispatched in parallel worktrees and merge cleanly.

### Frozen contracts (Track 0)

Before feature work begins, contracts land on `main` in a single early task and are **never edited from a feature track**:

- `src/contracts/zod/*` — every Zod schema (Profile, Pantry, Checkin, Recipe, RecommendationContext, MealCandidate). Shared by client, server, engine.
- `src/engine/ports/*` — `LlmClient`, `SignalProvider`, `Logger`.
- `src/server/contracts.ts` — Server Action signatures and Route Handler types.
- `src/db/schema/*.ts` — Drizzle table definitions.

### File-ownership rule

Every feature owns a folder slice across the layers:

```
pantry feature      → src/db/schema/pantry.ts
                      src/server/pantry/*
                      src/app/pantry/*
                      src/components/feature/pantry/*
profile feature     → src/db/schema/profile.ts
                      src/server/profile/*
                      src/app/profile/*
                      src/components/feature/profile/*
checkin feature     → … same shape
```

Cross-feature files (root layout, providers, tailwind config, tokens.css, `src/components/ui/*`) are owned by exactly one track. **No drive-by edits to shared files** from a feature branch.

### Parallelization waves

- **Wave 0:** Track 0 alone. Everything else blocks on it.
- **Wave 1 (massively parallel):** Tracks 1, 2, 4 in parallel worktrees.
- **Wave 2:** Tracks 3, 5, 6, 7, 11, 12 in parallel.
- **Wave 3:** Tracks 8, 9.
- **Wave 4:** Track 10 (composes everything; lots of `delight`/`bolder` polish).

### Subagent task contract

Every Cline Kanban card carries:

- **Spec/contract refs** — Zod schemas / port interfaces (paths + commit hash).
- **Owned paths** — explicit list; the agent commits to *only* touching these.
- **Acceptance criteria** — testable, narrow.
- **Definition of Done** — typecheck ✓ / lint ✓ / tests ✓ / no edits outside owned paths ✓ / code-review pass ✓.
- **Worktree command** — exact `git worktree add` invocation and branch name (e.g. `wt/track-5-pantry`).

### Cline Kanban setup

Board: `http://127.0.0.1:3484/whattoeat`. Following [Cline's core kanban workflow](https://docs.cline.bot/kanban/core-workflow):

- **Lanes:** `Backlog → Ready → In Progress → Review → Done`.
- **Card per track**, prefix `T0–T12`. Cards include contract refs, owned paths, acceptance criteria.
- **Wave label** on each card (`wave-0/1/2/3/4`).
- **Worktree branch name** in the card title.
- **Dependencies** field links blocking cards.
- A `scripts/seed-kanban.ts` (or documented manual setup if Cline lacks an API) builds cards from `docs/superpowers/specs/kanban-tasks.yaml`. **The kanban is regenerable from the repo**, never the source of truth itself.

### Worktree ergonomics

- `.gitignore` already isolates `.next/`, `node_modules/`, `.env*`.
- `scripts/wt.sh new <track-name>` creates `../whattoeat-<track>`, copies `.env.local`, runs `bun install`, picks a unique dev port.
- `CONTRIBUTING.md` (1 page) states the file-ownership rule + worktree workflow so any subagent prompt can reference it.

### Agent context bootstrap (auto)

`docs/PROJECT_RULES.md` is the **single source of truth** — one page, no fluff:

1. Mission in one paragraph.
2. Architecture rule: engine = pure, server = Supabase, app = UI.
3. File-ownership rule.
4. Kanban URL + workflow lanes.
5. Worktree workflow.
6. Definition of Done.
7. Forbidden patterns (raw hex in components, regex-parsing LLM output, drive-by edits to shared files, schema changes outside Track 0).
8. Where to find: spec, contracts, schemas, eval harness, kanban seed file.
9. Pre-push expectations.

**Auto-bootstrap shim files** (all generated from one template by `scripts/sync-agent-rules.ts`; pre-commit hook keeps them in sync):

- `CLAUDE.md` — Claude Code auto-loads.
- `AGENTS.md` — Codex / Cursor agents.
- `GEMINI.md` — Gemini CLI auto-loads.
- `.cursor/rules/project.mdc` — Cursor.
- `.github/copilot-instructions.md` — Copilot.

Each shim says: *"before doing anything, read `docs/PROJECT_RULES.md` and check the kanban for an unclaimed Ready card matching the current wave."*

### Slash commands committed under `.claude/commands/`

- **`/start-task`** — reads kanban, picks the next Ready card matching the agent's wave, creates a worktree, opens the spec section + contracts the card depends on, announces the plan before touching code.
- **`/finish-task`** — runs typecheck/lint/tests, dispatches `superpowers:code-reviewer` on the diff, moves the kanban card to Review on pass.

### Code review — two layers

**Layer 1 — during agent work (mandatory, part of DoD):**

- On `/finish-task`, `superpowers:code-reviewer` reviews the worktree diff against the spec, file-ownership rule, contracts, and the forbidden-patterns list.
- Verdict: `pass | request-changes | block`. Only `pass` advances the card to Review.
- Tasks touching `src/engine/` or `src/server/adapters/` additionally trigger an eval-harness run; the cost-vs-quality delta is posted to the card.
- Tasks touching `src/components/ui/*` additionally trigger a `polish` + `audit` skill pass.

**Layer 2 — at git push / PR (automated):**

- **Husky pre-commit** — `lint-staged` (typecheck on changed TS, biome lint, format).
- **Husky pre-push** — full typecheck, full lint, full test suite, fast diff-only review pass; blocks on failure.
- **GitHub Actions:**
  - `quality.yml` — build + typecheck + lint + tests.
  - `engine-eval.yml` — runs `bun run engine:eval` on PRs touching engine/adapter files; posts the eval table as a PR comment.
  - `review.yml` — runs the code-reviewer agent on PR diffs; required check before merge to `main`.
  - `security-review.yml` — runs `/security-review` on diffs touching auth, RLS, route handlers, or external integrations.
- **Branch protection on `main`:** all four checks + 1 human (or human-approved-agent) review.

### Bug-catching beyond review

- **Sentry** on the Next.js app (errors + perf, source maps). Tagged with `userId`, `track`, `engineVersion`, `promptsVersion`.
- Engine errors carry typed subclasses; route handlers map to status codes; `recommendation_runs` captures every failed run with the input snapshot — regressions reproducible from production data.
- A weekly scheduled agent (via `/schedule`) reads the last 7 days of `recommendation_runs` for low-quality outputs and opens triage cards on the kanban.

## 7. Core User Flows + LLM Pipeline

### Onboarding (3 panes)

1. Welcome + value prop, magic-link auth.
2. **Set goal** — `cut | maintain | bulk` SegmentedControl + height/weight/sex/activity-level. Server computes TDEE + macro split; the user can edit.
3. **Seed pantry** — curated 24-item starter grid (chicken, eggs, rice, etc.) + free-add. Tap to toggle.

### Pantry

- Chip grid grouped by category. Search bar on top, free-text add at bottom, voice add (Web Speech API — real this time, not the v1 hardcoded fake).
- Add flow: user types → server normalizes via LLM → confirmation chip ("Add **chicken breast**?") with 2s undo. Failed normalization falls back to literal input.
- Toggle `available` without deleting; long-press to remove.
- Bulk-add via paste (newline / comma separated); each line normalized.

### Profile

- Goal + macros (auto, editable), allergies, dislikes, cuisines (multi-select chips), equipment.
- "Connections" section with placeholder cards for Apple Health / Eight Sleep / Superpower marked **Coming soon** — visual hook for v2.1.

### Daily Check-in

- Vaul drawer triggered from Home. 3 taps: energy 1–5, training (none/light/hard), hunger (low/normal/high). Optional note. ~5s.
- Idempotent per `(user_id, date)`; second tap edits today's row.

### "Feed Me" — the centerpiece

1. Tap from Home (or pantry, or recipe-list empty state).
2. Optional override sheet: meal type + time budget (skippable; we infer from time of day).
3. Server Action assembles `RecommendationContext` → calls `recommend()` → streams to client via Route Handler.
4. UI staged reveal: button compresses → gradient bloom → 1–3 `MealCard`s stagger in → rationale fades in beneath.
5. Tap a card → expand to recipe view → "I cooked this" writes `cooked_log` and prompts for a 1-tap rating.
6. The recommendation persists to `recommendation_runs` *before* candidates are shown — every result is auditable and re-openable from history.

### Recipe view

- Title, why-line, time + servings, **`MacroRing`** with animated count-up, ingredients with pantry-coverage chip per item (have/missing), step-by-step instructions with optional timers per step.
- Sticky bottom bar: Save (heart) + "I cooked this".
- Recipes from `recommendation_runs` persist to `recipes` lazily — only on save or cook. Browsing doesn't pollute saved-recipes.

### LLM pipeline

**Two-call architecture:**

1. **Plan call** (cheap model): given context, produce N candidate concepts — `[{ title, oneLineWhy, cuisine, est_minutes, pantryFit }]`. Strict schema.
2. **Detail calls** (parallel, one per chosen candidate): expand to full recipe — ingredients/steps/macros. Strict schema.

This separation makes planning fast/cheap, lets us re-rank concepts before paying for full detail, and bounds blast radius on parser failures.

**Prompt structure (deterministic, cache-friendly):**

- **System prompt** — stable across users, ~1–2k tokens, explains schema, rules, brand voice for `oneLineWhy`. Cached via Gemini's prompt-cache.
- **User prompt** — deterministic JSON serialization of `RecommendationContext` (sorted keys, stable arrays). Same context → same prompt → cache-eligible.
- System prompt is versioned (`PROMPTS_VERSION = '2026-04-26.1'`) and stored on the run for traceability.

**Structured output:**

- Gemini's native JSON-mode / response-schema with our Zod schema reflected to JSON Schema (`zod-to-json-schema`).
- On schema validation fail: 1 retry with the validator error appended ("Your previous response failed validation here: …"). On second fail: throw `EngineParseError`. **No regex fallback.**

**Streaming:**

- Plan call streams concept tokens; UI shows skeleton cards filling in.
- Detail calls run in parallel after plan resolves; cards expand as each detail completes.

**Safety / sanitization (engine, post-LLM):**

- Hard filter against `profile.allergies`. If all candidates drop, throw `EngineSafetyError` (UI: "Couldn't find anything safe right now — try adding to pantry").
- Macro-bounds check ±30%; out-of-range candidates flagged with a soft chip but not dropped.

**Cost discipline:**

- Per-user soft cap on calls/day (default 30) tracked in Upstash Redis. Hit cap → friendly message, not 500.
- Cost + latency + tokens logged to `recommendation_runs` for every call.

**Model selection:**

- Eval harness picks the cheapest model that clears the rubric. Plan + Detail can be different models.

## 8. Auth, RLS, Observability, Cleanup

### Auth (Supabase)

- **Magic-link only.** No password forms.
- `@supabase/ssr` for server/client session sharing. Cookie-based, refreshed by middleware.
- `src/middleware.ts` enforces auth on `/(authenticated)/**`. Public routes: `/`, `/auth/*`, `/onboarding/*`.
- Server Components/Actions/Route Handlers fetch the user via `supabase.auth.getUser()` (verified, not `getSession()`). Never trust client-supplied user IDs.
- Account deletion path from day one (Profile → Danger zone).

### RLS

- Default-deny on every table.
- Standard policy on user-owned tables: `using (auth.uid() = user_id) with check (auth.uid() = user_id)` for all operations.
- `recommendation_runs` is read-only client-side (insert via service-role from Route Handler).
- `signal_snapshots` is read-only client-side; writes only from integration webhooks (later) using service-role + verified provider signature.
- **RLS tests required.** Track 0 ships a Vitest suite that authenticates as user A, attempts to read user B's rows, asserts denial. CI gates on this.

### Rate limiting (Upstash Redis)

- `/api/recommend`: 30/day per user (soft cap), 10/hour burst.
- LLM-normalize calls (pantry add): 200/day per user.
- Auth magic-link: 5/15min per IP + email.
- 429 with friendly error and retry-after.

### Observability

- **Sentry** Next.js integration — errors, performance, source maps. Events tagged with `userId`, `track`, `engineVersion`, `promptsVersion`.
- **Structured logs** via `pino`. Request-scoped logger via Next.js `headers()` / `AsyncLocalStorage`. Always-present fields: `req_id`, `user_id` (if known), `route`.
- `recommendation_runs` is the *primary* observability surface for engine quality.
- Vercel Analytics for traffic. Web Vitals piped to Sentry.
- `/admin/health` (only the project owner's email) — last 50 runs, current eval-harness leaderboard, daily cost.

### Error contract (end-to-end)

- **Engine** throws typed `EngineError` subclasses.
- **Server services** wrap engine calls and surface typed `ServerError` to Server Actions/Route Handlers.
- **Route Handlers** map to status codes; payload is `{ error: { code, message } }` — never raw exception messages.
- **UI** has a feature-level error boundary on the recommendation flow with a "try again" CTA; global `error.tsx` for unhandled cases. Sonner toast for non-fatal Server Action errors.

### Secrets / env

- One `.env.example` checked in. Real `.env.local` ignored. Vercel Project env for prod.
- Supabase service-role key only used in Route Handlers / Server Actions. ESLint rule pins this.
- Env surface for v2.0: Supabase URL/anon/service-role, Gemini key, Upstash creds, Sentry DSN, app URL.
