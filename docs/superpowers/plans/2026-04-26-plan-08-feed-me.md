# Plan 08 — Feed Me Flow + Recommendation Result UI

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire the end-to-end "Feed Me" recommendation surface — build the `RecommendationContext` server-side from live profile, pantry, check-in, and cook-history data; invoke the engine (real or fake, selected by env); enforce the daily rate-limit cap before every LLM call; and render 1–3 `MealCard` components with save, cook, and regenerate actions. Every engine error variant (`EngineSafetyError`, `EngineParseError`, `AllergenViolation`) surfaces as a designed empty state rather than a crash. `withInstrumentation` from Track 12 wraps the regenerate action and the engine span. The critical cross-track contract is that this track reads — never writes — the outputs of T2 (engine), T3 (Gemini adapter), T5 (pantry), T6 (profile), T7 (check-in), and T9 (recipes).

**Architecture:** Two thin integration layers beneath the page. (1) `src/lib/feed-me/buildContext.ts` — a pure async assembler that calls repo helpers from T5/T6/T7/T9 and shapes their outputs into a `RecommendationContext`. It receives a pre-authenticated Supabase client so it is independently testable with mocked clients. (2) `src/server/recommendation/actions.ts` — Server Actions that call `assertWithinDailyCap()`, build the context, resolve the `LlmClient` from the environment, invoke `recommend()`, persist the run to `recommendation_runs`, and return a typed `ActionResult`. The page itself is a React Server Component that renders the shell and falls through to client islands for interactive mutations (save / cook / regenerate). All state is URL-search-param-free; the React state owned by the `FeedMeIsland` client component is the authoritative source during a session, flushed to the server on every mutation.

**Why a server component page, not a route handler?** The spec calls for a "staged reveal motion" — the outer shell (header, empty-state fallback, skeleton) streams instantly from the RSC; the recommendation data populates via a client-side server action call triggered on mount. This hybrid (RSC shell + client action trigger) avoids the complexity of streaming route handlers while still keeping sensitive context assembly (profile macros, allergen list) on the server where it belongs.

**Tech Stack:** Bun, Vitest, Zod, `@supabase/ssr`, Next.js 15 App Router (Server Components + `'use server'` actions), React 19 (`useOptimistic`, `useTransition`, `useActionState`), design-system primitives from Track 1 (`MealCard`, `Button`, `Skeleton`, `Toast`, `Sheet`), `withInstrumentation` from T12, `assertWithinDailyCap` from T12.

**Spec reference:** `docs/superpowers/specs/2026-04-26-whattoeat-2.0-design.md` — sections 2 (architecture / hexagonal engine / file-ownership rule), 3 (data model: `recommendation_runs`), 4 (Decision Engine Contract, `RecommendationContext`, `MealCandidate`), 7 (Feed Me flow: context assembly, staged reveal, MealCard, rate-limit 429 message), 8 (RLS, observability / Sentry tagging, rate limit / assertWithinDailyCap).

## Conventions used in this plan

- All file paths are repo-relative; bash commands use the absolute root `/Users/ravishah/Documents/whattoeat`.
- `bun` is the package manager and test runner (`bun run test`, `bun run typecheck`, `bun run lint`).
- All imports use the `@/` alias for `src/` (e.g. `@/lib/feed-me`, `@/server/recommendation`, `@/components/ui/button`).
- Commit message prefixes:
  - `feed-me:` — source files under `src/lib/feed-me/`, `src/server/recommendation/`, `src/app/(authenticated)/feed-me/`
  - `feed-me-test:` — test files under `src/lib/feed-me/__tests__/`, `src/server/recommendation/__tests__/`
  - `feed-me-ui:` — files under `src/components/feature/feed-me/`
- **TDD discipline:** every test task runs first (expected RED), then the implementation task makes it GREEN. Steps are annotated `— expected RED` or `— expected GREEN`.
- **Rate-limit discipline:** `assertWithinDailyCap()` is called at the top of the `regenerate` server action, before `buildContext` or the engine. If the cap is hit, the action returns `{ ok: false, error: { code: 'RATE_LIMITED', … } }` immediately — the engine is never invoked.
- **Error discipline:** `EngineResult<RecommendationResult>` is a discriminated union (`ok: true | ok: false`). The server action never throws; it always returns `ActionResult<RecommendationResult>`. The `FeedMeIsland` client component branches on `ok` and renders the appropriate `EmptyState` variant.
- **Optimistic discipline:** the "Save" and "Cook" buttons use `useOptimistic` to update card chrome instantly and roll back on a `ok: false` server response. The recommendation list itself is NOT optimistic — it only updates once the server action returns the full engine result.
- **RLS discipline:** every DB write (insert `recommendation_runs`) uses the service-role Drizzle client because `recommendation_runs` rows are written on behalf of the user after the engine completes — they are not user-initiated mutations and therefore flow through service-role, not the anon RLS client. All reads (pantry, profile, check-in, recipes) use the RLS-anon client threaded from `requireUser()`, exactly as described in the T5/T6/T7 plans.
- **No client-supplied user IDs:** every server action calls `requireUser()` to source `userId`. Nothing from the request body is trusted for identity.
- All TS/JSX uses literal angle brackets (no HTML entities). Apostrophes in strings use the ASCII single-quote character.

```ts
/**
 * buildContext — assembles a RecommendationContext from live data sources.
 *
 * Reads from:
 *   - src/server/pantry   (T5)
 *   - src/server/profile  (T6)
 *   - src/server/checkin  (T7)
 *   - src/server/recipes  (T9 — getRecentCookTitles)
 *
 * Returns a discriminated-union result: { ok: true; value: RecommendationContext }
 * or { ok: false; error: BuildContextError }.
 *
 * NO Next.js imports. NO network calls beyond what the helpers do internally.
 * This function is pure in the sense that all I/O is injected via the helpers
 * (already imported as module-level references) and the SupabaseClient arg.
 */

import type { SupabaseClient } from '@supabase/supabase-js';
import type { RecommendationContext } from '@/contracts/zod/recommendation';

import { listActivePantryItems } from '@/server/pantry/index';
import { getMyProfile, computeTargets } from '@/server/profile/index';
import { getTodayCheckin } from '@/server/checkin/index';
import { getRecentCookTitles } from '@/server/recipes/actions';

// ---------------------------------------------------------------------------
// Error type local to this module
// ---------------------------------------------------------------------------

export interface BuildContextError {
  /** Machine-readable code for EmptyState variant selection. */
  code: 'PROFILE_INCOMPLETE' | 'DATA_FETCH_FAILED';
  message: string;
}

export type BuildContextResult =
  | { ok: true; value: RecommendationContext }
  | { ok: false; error: BuildContextError };

// ---------------------------------------------------------------------------
// Options
// ---------------------------------------------------------------------------

export interface BuildContextOptions {
  /**
   * Number of meal candidates to request from the engine.
   * Defaults to 3 (the spec-mandated maximum surface).
   */
  candidateCount?: number;
  /**
   * Client local date string (YYYY-MM-DD) passed to getTodayCheckin() to avoid
   * UTC offset mismatches. If omitted, getTodayCheckin falls back to UTC date.
   */
  localDate?: string;
}

// ---------------------------------------------------------------------------
// Assembler
// ---------------------------------------------------------------------------

const RECENCY_WINDOW_DAYS = 7;

export async function buildContext(
  supabase: SupabaseClient,
  userId: string,
  opts: BuildContextOptions = {},
): Promise<BuildContextResult> {
  const candidateCount = opts.candidateCount ?? 3;

  try {
    // Fetch all data sources in parallel for minimum latency.
    const [profile, pantryAll, checkin, recentCookTitles] = await Promise.all([
      getMyProfile(supabase, userId),
      listActivePantryItems(supabase, userId),
      getTodayCheckin(supabase, userId, opts.localDate),
      getRecentCookTitles(
        // getRecentCookTitles calls requireUser() internally; the `since`
        // argument is derived from the recency window defined above.
        new Date(Date.now() - RECENCY_WINDOW_DAYS * 24 * 60 * 60 * 1000).toISOString(),
      ),
    ]);

    // Profile is mandatory — without it the engine cannot produce safe output.
    if (!profile) {
      return {
        ok: false,
        error: {
          code: 'PROFILE_INCOMPLETE',
          message:
            'Your profile is incomplete. Please fill in your goals, weight, and height before using Feed Me.',
        },
      };
    }

    // Targets can be null if biometric fields are missing even when a profile row exists.
    const targets = profile.targets ?? computeTargets(profile);
    if (!targets) {
      return {
```

```ts
 * resolveClient — returns the appropriate LlmClient based on the runtime environment.
 *
 * In production (RECOMMEND_LLM_CLIENT=gemini or unset):
 *   - Instantiates GeminiLlmClient with GEMINI_API_KEY, GEMINI_MODEL_QUALITY.
 *   - Falls back to GEMINI_MODEL_CHEAP if the env var is absent.

 *   - Returns FakeLlmClient with a configurable response fixture.

import { FakeLlmClient } from '@/engine/index';
import { GeminiLlmClient } from '@/server/adapters/gemini-llm';

  const selector = process.env.RECOMMEND_LLM_CLIENT ?? 'gemini';

  if (selector === 'fake') {
    // FakeLlmClient is safe to return without configuration; it returns a
    // hardcoded successful result. Tests that need specific outputs should
    // pass a pre-configured instance via `override`.
    return new FakeLlmClient();

    process.env.GEMINI_MODEL_CHEAP ??
```

```ts
/**
 * Recommendation runs repo — persists a record of every engine invocation.
 *
 * Uses the Drizzle service-role client (not the RLS-anon client) because:
 *   - The insert happens after the engine call completes, not in response to a
 *     direct user mutation.
 *   - The RLS policy on recommendation_runs is `SELECT` only for the anon role;
 *     INSERT is granted to service_role.
 *
 * Callers (server actions) are responsible for passing the correct userId.
 * The repo does not call requireUser().
 */

import { db } from '@/db/client';
import { recommendationRuns } from '@/db/schema/recommendation-runs';
import type { RecommendationContext } from '@/contracts/zod/recommendation';
import type { MealCandidate } from '@/contracts/zod/recommendation';

export interface InsertRunArgs {
  userId: string;
  /** The full context passed to the engine. Stored as JSONB for debugging. */
  context: RecommendationContext;
  /**
   * The engine result candidates. Null when the engine returned an error
   * (we still record the run for rate-limit accounting).
   */
  candidates: MealCandidate[] | null;
  /** Engine error code if the run failed. Null on success. */
  errorCode: string | null;
  /** Prompt template version from the engine (PROMPTS_VERSION). */
  promptsVersion: string;
  /** Wall-clock time for the full engine call in milliseconds. */
  durationMs: number;
}

export async function insertRecommendationRun(args: InsertRunArgs): Promise<string> {
  const [row] = await db
    .insert(recommendationRuns)
    .values({
      user_id: args.userId,
      context: args.context as Record<string, unknown>,
      candidates: args.candidates as Record<string, unknown>[] | null,
      error_code: args.errorCode,
      prompts_version: args.promptsVersion,
      duration_ms: args.durationMs,
      created_at: new Date().toISOString(),
    })
    .returning({ id: recommendationRuns.id });

  if (!row) {
    throw new Error('insertRecommendationRun: Drizzle returned no rows — check schema types.');
  }

  return row.id;
}
```

### Task 7: Implement `src/server/recommendation/actions.ts` (GREEN)

**Files:** `src/server/recommendation/actions.ts`

The `regenerateAction` Server Action is the heart of this track. Its responsibilities, in strict order:

1. Call `requireUser()` to get `userId` — abort with a redirect to `/auth/login` if unauthenticated.
2. Call `assertWithinDailyCap(userId)` from T12 — return `RATE_LIMITED` immediately if exhausted. No engine call.
3. Call `buildContext(supabase, userId, opts)` — return `PROFILE_INCOMPLETE` or `DATA_FETCH_FAILED` immediately if context assembly fails.
4. Call `resolveClient()` to get the `LlmClient`.
5. Call `recommend(ctx, { llm: client, recentCookTitles: ctx.recentCookTitles })` and record wall-clock time.
6. Call `insertRecommendationRun()` with the result (success or error).
7. Return `ActionResult<RecommendationResult>` to the client.

`withInstrumentation` from T12 wraps steps 4–6 as a single engine span, providing Sentry tagging and latency measurement.

```ts
'use server';

/**
 * Feed Me server actions.
 *
 * regenerateAction — the primary action. Assembles context, enforces the daily
 *   cap, runs the engine, persists the run, and returns typed candidates.
 *
 * All actions call requireUser() first. No userId may come from request bodies.
 */

import { redirect } from 'next/navigation';
import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

import { requireUser } from '@/server/auth/index';
import { assertWithinDailyCap } from '@/lib/rate-limit/index';
import { buildContext, type BuildContextOptions } from '@/lib/feed-me/buildContext';
import { resolveClient } from '@/lib/feed-me/resolveClient';
import { recommend } from '@/engine/index';
import { EngineSafetyError, EngineParseError } from '@/engine/errors';
import { withInstrumentation } from '@/server/instrumentation/index';
import { insertRecommendationRun } from '@/server/recommendation/repo';
import { PROMPTS_VERSION } from '@/lib/version';

import type { LlmClient } from '@/engine/ports/llm';
import type { MealCandidate, RecommendationContext } from '@/contracts/zod/recommendation';

// ---------------------------------------------------------------------------
// Return type
// ---------------------------------------------------------------------------

export type RecommendActionSuccess = {
  runId: string;
  candidates: MealCandidate[];
  context: RecommendationContext;
};

export type RecommendActionError = {
  code:
    | 'RATE_LIMITED'
    | 'PROFILE_INCOMPLETE'
    | 'DATA_FETCH_FAILED'
    | 'ENGINE_SAFETY'
    | 'ENGINE_PARSE'
    | 'ENGINE_UNKNOWN';
  message: string;
  /** ISO timestamp — only present for RATE_LIMITED */
  resetAt?: string;
};

export type RecommendActionResult =
  | { ok: true; value: RecommendActionSuccess }
  | { ok: false; error: RecommendActionError };

// ---------------------------------------------------------------------------
// Supabase server client factory
// ---------------------------------------------------------------------------

async function makeServerClient() {
  const cookieStore = await cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll: () => cookieStore.getAll(),
        setAll: (toSet) => {
          for (const { name, value, options } of toSet) {
            cookieStore.set(name, value, options);
          }
        },
      },
    },
  );
}

// ---------------------------------------------------------------------------
// Inner engine span — wrapped by withInstrumentation in the exported action
// ---------------------------------------------------------------------------

async function _runEngineSpan(
  ctx: RecommendationContext,
  llm: LlmClient,
  userId: string,
): Promise<RecommendActionResult> {
  const start = performance.now();
  let candidates: MealCandidate[] | null = null;
  let errorCode: string | null = null;

  try {
    const engineResult = await recommend(ctx, {
      llm,
      recentCookTitles: ctx.recentCookTitles,
      timeoutMs: 30_000,
    });

    const durationMs = Math.round(performance.now() - start);

    if (!engineResult.ok) {
      const err = engineResult.error;
      if (err instanceof EngineSafetyError) {
        errorCode = 'ENGINE_SAFETY';
      } else if (err instanceof EngineParseError) {
        errorCode = 'ENGINE_PARSE';
      } else {
        errorCode = 'ENGINE_UNKNOWN';
      }

      const runId = await insertRecommendationRun({
        userId,
        context: ctx,
        candidates: null,
        errorCode,
        promptsVersion: PROMPTS_VERSION,
        durationMs,
      });

      return {
        ok: false,
        error: {
          code: errorCode as RecommendActionError['code'],
          message:
            errorCode === 'ENGINE_SAFETY'
              ? 'All meal suggestions were filtered out — likely due to your allergen settings. Try updating your pantry or relaxing a preference.'
              : 'The recommendation engine returned an unexpected response. Try again in a moment.',
        },
      };
    }

    candidates = engineResult.value.candidates;

    const runId = await insertRecommendationRun({
      userId,
      context: ctx,
      candidates,
      errorCode: null,
      promptsVersion: PROMPTS_VERSION,
      durationMs,
    });

    return {
      ok: true,
      value: {
        runId,
        candidates,
        context: ctx,
      },
    };
  } catch (err) {
    const durationMs = Math.round(performance.now() - start);

    // Unexpected throw — log and persist with UNKNOWN code.
    await insertRecommendationRun({
      userId,
      context: ctx,
      candidates: null,
      errorCode: 'ENGINE_UNKNOWN',
      promptsVersion: PROMPTS_VERSION,
      durationMs,
    }).catch(() => {
      // Swallow secondary failure to avoid masking the primary error.
    });

    throw err; // Re-throw so withInstrumentation can capture via Sentry.
  }
}

// ---------------------------------------------------------------------------
// Exported server action
// ---------------------------------------------------------------------------

export async function regenerateAction(
  opts?: BuildContextOptions & { llmOverride?: LlmClient },
): Promise<RecommendActionResult> {
  // Step 1: Auth
  let userId: string;
  try {
    const user = await requireUser();
    userId = user.userId;
  } catch {
    redirect('/auth/login');
  }

  // Step 2: Rate-limit check — must happen before any expensive I/O.
  const rateLimitResult = await assertWithinDailyCap(userId);
  if (!rateLimitResult.ok) {
    return {
      ok: false,
      error: {
        code: 'RATE_LIMITED',
        message: `You have reached your daily recommendation limit. Your cap resets at ${rateLimitResult.error.resetAt ?? 'midnight UTC'}.`,
        resetAt: rateLimitResult.error.resetAt,
      },
    };
  }

  // Step 3: Build context
  const supabase = await makeServerClient();
  const contextResult = await buildContext(supabase, userId, {
    candidateCount: opts?.candidateCount,
    localDate: opts?.localDate,
  });

  if (!contextResult.ok) {
    return {
      ok: false,
      error: {
        code: contextResult.error.code as RecommendActionError['code'],
        message: contextResult.error.message,
      },
    };
  }

  const ctx = contextResult.value;

  // Step 4: Resolve LLM client (or use override for testing)
  const llm = opts?.llmOverride ?? resolveClient();

  // Step 5-6: Run engine + persist run, wrapped in instrumentation for Sentry tagging.
  const instrumentedSpan = withInstrumentation(
    'engine.recommend',
    () => _runEngineSpan(ctx, llm, userId),
    { userId },
  );

  return instrumentedSpan();
}
```

### Task 9: `MealCard` + `MealCardStack` components

**Files:** `src/components/feature/feed-me/MealCard.tsx`, `src/components/feature/feed-me/MealCardStack.tsx`

`MealCard` is a client component that renders a single `MealCandidate` with save and cook action buttons. The save button calls the T9 `saveRecipe` server action; the cook button calls `markCooked`. Both use `useOptimistic` to provide instant visual feedback: the button disables and its icon animates to a spinner while the server action is in-flight, then reverts to its default state on error or advances to the confirmed state on success. A toast notification (from `@/components/ui/toast`) summarises the outcome.

`MealCardStack` is a presentational wrapper that orders the cards and adds the staggered reveal animation described in the spec. The stack does not communicate with the server directly — it delegates to individual `MealCard` instances.

```tsx
'use client';

import { useOptimistic, useTransition, useState } from 'react';
import { Bookmark, ChefHat, Loader2 } from 'lucide-react';
import { Button } from '@/components/ui/button';
import { MealCard as MealCardPrimitive } from '@/components/ui/meal-card';
import { toast } from '@/components/ui/toast';
import { saveRecipe, markCooked } from '@/server/recipes/actions';
import type { MealCandidate } from '@/contracts/zod/recommendation';

interface MealCardProps {
  candidate: MealCandidate;
  /** 0-based index used for the staggered reveal delay. */
  index: number;
}

type CardState = 'idle' | 'saving' | 'saved' | 'cooking' | 'cooked';

export function MealCard({ candidate, index }: MealCardProps) {
  const [state, setState] = useState<CardState>('idle');
  const [optimisticState, setOptimisticState] = useOptimistic<CardState>(state);
  const [, startTransition] = useTransition();

  const staggerDelayMs = index * 80;

  async function handleSave() {
    setOptimisticState('saving');
    startTransition(async () => {
      const result = await saveRecipe(candidate);
      if (result.ok) {
        setState('saved');
        toast.success(`"${candidate.title}" saved to your recipes.`);
      } else {
        setState('idle');
        toast.error(`Could not save — ${result.error.message}`);
      }
    });
  }

  async function handleCook() {
    setOptimisticState('cooking');
    startTransition(async () => {
      const result = await markCooked(candidate);
      if (result.ok) {
        setState('cooked');
        toast.success(`Logged "${candidate.title}" as cooked!`);
      } else {
        setState('idle');
        toast.error(`Could not log — ${result.error.message}`);
      }
    });
  }

  const isSaving = optimisticState === 'saving';
  const isCooking = optimisticState === 'cooking';
  const isSaved = optimisticState === 'saved';
  const isCooked = optimisticState === 'cooked';

  return (
    <div
      className="animate-in fade-in slide-in-from-bottom-4"
      style={{ animationDelay: `${staggerDelayMs}ms`, animationFillMode: 'both' }}
    >
      <MealCardPrimitive
        title={candidate.title}
        oneLineWhy={candidate.oneLineWhy}
        cuisine={candidate.cuisine ?? undefined}
        estMinutes={candidate.totalMinutes}
        macros={candidate.macros}
        missingItems={candidate.missingItems}
        pantryCoverage={candidate.pantryCoverage}
        actions={
          <div className="flex gap-2">
            <Button
              size="sm"
              variant={isSaved ? 'secondary' : 'outline'}
              disabled={isSaving || isSaved || isCooked}
              onClick={handleSave}
              aria-label={isSaved ? 'Saved' : 'Save recipe'}
            >
              {isSaving ? (
                <Loader2 className="h-4 w-4 animate-spin" />
              ) : (
                <Bookmark className={`h-4 w-4 ${isSaved ? 'fill-current' : ''}`} />
              )}
              <span className="ml-1.5">{isSaved ? 'Saved' : 'Save'}</span>
            </Button>

            <Button
              size="sm"
              variant={isCooked ? 'secondary' : 'default'}
              disabled={isCooking || isCooked}
              onClick={handleCook}
              aria-label={isCooked ? 'Cooked!' : 'I cooked this'}
            >
              {isCooking ? (
                <Loader2 className="h-4 w-4 animate-spin" />
              ) : (
                <ChefHat className="h-4 w-4" />
              )}
              <span className="ml-1.5">{isCooked ? 'Cooked!' : 'I cooked this'}</span>
            </Button>
          </div>
        }
      />
    </div>
  );
```

### Task 11: `FeedMeIsland` root client component

**Files:** `src/components/feature/feed-me/FeedMeIsland.tsx`

`FeedMeIsland` is the root client island. It is hydrated server-side with no initial data and immediately triggers `regenerateAction` on mount via `useEffect`. This "fetch on mount" pattern keeps the RSC page shell lightweight (no Suspense waterfall) and gives the server component a chance to stream the skeleton frame before the engine call begins.

State machine: `idle` → `loading` (engine call in flight) → `success` (candidates displayed) or one of the error states. The `RegenerateButton` transitions `success` back to `loading`, replacing the stack with the skeleton and re-firing the action.

The island also accepts an optional `localDate` prop from the server (injected by the page) so check-in timezone resolution works correctly.

- [ ] **Step 11.1 — Write `src/components/feature/feed-me/FeedMeIsland.tsx`**

```tsx
'use client';

import { useEffect, useState, useTransition } from 'react';
import { SkeletonStack } from './SkeletonStack';
import { MealCardStack } from './MealCardStack';
import { EmptyState, type EmptyStateVariant } from './EmptyState';
import { RegenerateButton } from './RegenerateButton';
import { regenerateAction, type RecommendActionResult } from '@/server/recommendation/actions';
import type { MealCandidate } from '@/contracts/zod/recommendation';

interface FeedMeIslandProps {
  /** Client's local date string (YYYY-MM-DD) passed from the RSC page. */
  localDate: string;
}

type IslandPhase =
  | { phase: 'loading' }
  | { phase: 'success'; candidates: MealCandidate[]; runId: string }
  | { phase: 'error'; variant: EmptyStateVariant; resetAt?: string };

export function FeedMeIsland({ localDate }: FeedMeIslandProps) {
  const [islandState, setIslandState] = useState<IslandPhase>({ phase: 'loading' });
  const [isPending, startTransition] = useTransition();

  async function runAction() {
    setIslandState({ phase: 'loading' });

    startTransition(async () => {
      let result: RecommendActionResult;
      try {
        result = await regenerateAction({ localDate });
      } catch {
        setIslandState({
          phase: 'error',
          variant: 'data-fetch-failed',
        });
        return;
      }

      if (result.ok) {
        setIslandState({
          phase: 'success',
          candidates: result.value.candidates,
          runId: result.value.runId,
        });
      } else {
        const errCode = result.error.code;
        const variantMap: Record<typeof errCode, EmptyStateVariant> = {
          RATE_LIMITED: 'rate-limited',
          PROFILE_INCOMPLETE: 'profile-incomplete',
          DATA_FETCH_FAILED: 'data-fetch-failed',
          ENGINE_SAFETY: 'engine-safety',
          ENGINE_PARSE: 'engine-parse',
          ENGINE_UNKNOWN: 'engine-parse', // surface as generic parse error
        };
        setIslandState({
          phase: 'error',
          variant: variantMap[errCode] ?? 'data-fetch-failed',
          resetAt: result.error.resetAt,
        });
      }
    });
  }

  // Fire on mount — no stale-while-revalidate needed; engine results are ephemeral.
  useEffect(() => {
    runAction();
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, []);

  // -------------------------------------------------------------------------
  // Render
  // -------------------------------------------------------------------------

  if (islandState.phase === 'loading' || isPending) {
    return (
      <div className="w-full">
        <SkeletonStack />
      </div>
    );
  }

  if (islandState.phase === 'error') {
    return (
      <EmptyState
        variant={islandState.variant}
        resetAt={islandState.resetAt}
        onRetry={runAction}
      />
    );
  }

  // phase === 'success'
  return (
    <div className="w-full space-y-6">
      <MealCardStack candidates={islandState.candidates} />

      <div className="flex justify-center pb-8">
        <RegenerateButton onRegenerate={runAction} isPending={isPending} />
      </div>
    </div>
  );
}
```

**Files:** `src/app/(authenticated)/feed-me/page.tsx`, `src/app/(authenticated)/feed-me/loading.tsx`, `src/app/(authenticated)/feed-me/error.tsx`

**Timezone note:** The most reliable way to pass the client's local date to the server is via a cookie set by a small client-side script on first load. For the initial implementation we use the `tz` cookie if present, falling back to UTC. Track 10 (Home) is responsible for injecting the cookie-setter script on the home page before the user navigates to `/feed-me`. The fallback is acceptable because most users are checked in during daytime UTC+0 to UTC+14.

The `loading.tsx` file renders `SkeletonStack` as the Next.js Suspense fallback — this shows before the RSC even renders, giving the fastest-possible first paint.

The `error.tsx` file is the App Router error boundary for this route segment. It renders `EmptyState` with `variant="data-fetch-failed"` and passes the error message to Sentry.

```tsx
import { cookies } from 'next/headers';
import { FeedMeIsland } from '@/components/feature/feed-me/FeedMeIsland';

export const metadata = {
  title: 'Feed Me — WhatToEat',
  description: 'Get personalised meal recommendations based on your pantry and goals.',
};

// The page RSC does not call requireUser() — the middleware at src/middleware.ts
// already guarantees that only authenticated users reach /(authenticated)/*.
// If middleware is bypassed somehow, the island's regenerateAction will redirect.

export default async function FeedMePage() {
  const cookieStore = await cookies();

  // Derive the client's local date from the 'tz' cookie if available.
  // The cookie value is the UTC-offset string set by the client-side timezone
  // detector (shipped in Track 10). Falls back to UTC today.
  const tzOffset = cookieStore.get('tz')?.value;
  const localDate = tzOffset
    ? getLocalDateFromOffset(tzOffset)
    : new Date().toISOString().slice(0, 10);

  return (
    <main className="min-h-screen bg-background">
      <div className="sticky top-0 z-10 bg-background/80 backdrop-blur-sm border-b border-border px-4 py-3">
        <h1 className="text-lg font-semibold text-foreground">Feed Me</h1>
      </div>

      <div className="py-6">
        <FeedMeIsland localDate={localDate} />
      </div>
    </main>
  );

// ---------------------------------------------------------------------------
// Helpers
// ---------------------------------------------------------------------------

/**
 * Derives today's local date string (YYYY-MM-DD) from a UTC offset string.
 * The offset format is '+HH:MM' or '-HH:MM' as produced by the browser's
 * `Intl.DateTimeFormat().resolvedOptions().timeZone` processing.
 *
 * Falls back to UTC today if the offset is malformed.
 */
function getLocalDateFromOffset(tzOffset: string): string {
  try {
    const sign = tzOffset.startsWith('-') ? -1 : 1;
    const [hourStr, minStr] = tzOffset.replace(/^[+-]/, '').split(':');
    const offsetMinutes = sign * ((parseInt(hourStr ?? '0', 10) * 60) + parseInt(minStr ?? '0', 10));
    const now = new Date();
    const localMs = now.getTime() + offsetMinutes * 60_000;
    return new Date(localMs).toISOString().slice(0, 10);
  } catch {
    return new Date().toISOString().slice(0, 10);
  }
}
```
